## Context

**Objective**: Implement immutable `Beschreibung` value object following DDD principles (ADR-0021), immutability pattern (ADR-0010), and onion architecture (ADR-0004) with validation (max 500 chars, non-blank) and XSS prevention via sanitization delegation.

**Initial State**: No Beschreibung value object existed. Pattern analysis identified 7 existing value objects (Produktname, Produktnummer, Produktfarbe, etc.) with varying implementation patterns - some using `private var` (Produktname - technical debt), others using `private val` (Produktfarbe - correct pattern).

**Target Architecture**:
- Domain layer value object (domain/model/produkte/produkt package)
- Immutable data class with `@ValueObject` annotation
- Factory method pattern for sanitization
- Fail-fast validation with domain exceptions
- 100/100 quality standard compliance

## Problem

**Primary Challenge**: Implement Beschreibung value object with correct immutability pattern while existing codebase showed conflicting patterns.

**Key Decisions Required**:
1. **Immutability Pattern**: Use `private var` (Produktname pattern) vs `private val` (Produktfarbe pattern)?
2. **Factory vs Constructor**: Public constructor with init block vs private constructor with factory method?
3. **Validation Order**: When to validate length - before or after sanitization?
4. **Sanitization Edge Case**: What if sanitization removes all content (e.g., HTML-only input)?

**Technical Constraints**:
- Kotlin 2.2.0 private constructor warning about copy() visibility
- Gradle environment error (error code 25) blocking test execution
- ADR-0010 mandates immutability (val properties only)
- ArchUnit ValueObjectTest enforces final fields

  ```kotlin
  @ValueObject
  data class Beschreibung private constructor(
    private val value: String  // IMMUTABLE (private val)
  ) {
    fun asString(): String = value

    companion object {
      private const val MAX_LENGTH = 500

      fun create(rawValue: String): Beschreibung {
        if (rawValue.isBlank()) throw BeschreibungIstLeerException(...)
        val sanitized = rawValue.sanitize()
        if (sanitized.isBlank()) throw BeschreibungIstLeerException(...)  // Edge case
        if (sanitized.length > MAX_LENGTH) throw BeschreibungIstZuLangException(...)
        return Beschreibung(sanitized)
      }
    }
  }
  ```

### Architecture Decisions

**Decision 1: Immutable Pattern with Factory Method**
- **Chosen**: `private val value` + factory method `create()`
- **Rationale**:
  - ADR-0010 mandates immutability (val only)
  - ArchUnit ValueObjectTest enforces final fields
  - Architecture-advisor REJECTED `private var` pattern as non-compliant
  - Factory method enables sanitization before construction

**Decision 2: Validation Order**
- **Chosen**: isBlank() → sanitize() → isBlank() → length check
- **Rationale**:
  - Fail-fast on obviously invalid input (empty string)
  - Sanitize to remove XSS threats
  - Re-validate after sanitization (edge case: HTML-only becomes empty)
  - Check length on sanitized value (prevents length bypass via HTML)

**Decision 3: German Exception Messages**
- **Chosen**: All exception names and messages in German
- **Rationale**: Ubiquitous language from ADR-0021, existing pattern consistency

**Decision 4: MAX_LENGTH Constant**
- **Chosen**: `private const val MAX_LENGTH = 500` in companion object
- **Rationale**: Refactoring-advisor identified magic number as code quality issue (DRY violation)

## Patterns Discovered

### Value Object Factory Pattern
```kotlin
companion object {
  private const val MAX_LENGTH = 500

  fun create(rawValue: String): ValueObject {
    // 1. Pre-validation (fail-fast)
    if (rawValue.isBlank()) throw ...

    // 2. Sanitization (security)
    val sanitized = rawValue.sanitize()

    // 3. Post-sanitization validation (edge cases)
    if (sanitized.isBlank()) throw ...

    // 4. Business rule validation
    if (sanitized.length > MAX_LENGTH) throw ...

    // 5. Construction with validated, sanitized value
    return ValueObject(sanitized)
  }
}
```

**Benefits**:
- Clear separation: validation → sanitization → construction
- Immutability guaranteed (`private val`)
- Edge case handling (sanitization side effects)
- Testable validation logic

### Strikt Assertion Patterns
```kotlin
// Nullable exception messages require !! operator
expectThat(exception.message!!).isEqualTo("expected message")

// Simple value assertions
expectThat(valueObject.asString()).isEqualTo(expectedValue)

// Exception type assertions
expectThrows<CustomException> { /* action */ }
```

### Test Organization Pattern
```kotlin
@Tag("unit")
internal class ValueObjectTest {
  // 1. Happy path tests
  @Test internal fun `valid creation`() { ... }

  // 2. Validation/exception tests
  @Test internal fun `empty rejection`() { ... }
  @Test internal fun `length limit`() { ... }

  // 3. Security tests
  @Test internal fun `XSS prevention`() { ... }

  // 4. Edge cases
  @Test internal fun `HTML-only input`() { ... }

  // 5. Data class behavior
  @Test internal fun `equality`() { ... }
}
```

4. **Test Duplication**: Initial implementation had duplicate equality test
   - **Impact**: Minor - caught by refactoring-advisor
   - **Prevention**: Better test planning in quality-assurance-expert phase

1. **❌ Using `private var` for Immutability**: Pattern-analyzer identified Produktname as technical debt, preventing replication
2. **❌ Magic Number 500**: Refactoring-advisor caught hardcoded limit before code review
