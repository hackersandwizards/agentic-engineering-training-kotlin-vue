## Solution

### Architecture Decisions

**Pattern Choice**: Nullable Field with Null-Safe Mapping
- **Rationale**:
  - Backward compatibility with existing MongoDB documents
  - Follows established pattern from Produktname, ProduktbildUrl, Produktfarbe
  - Enables gradual migration without data loss
  - Clean separation between domain (Beschreibung) and persistence (String)

**Key Design Elements**:
1. **Produkt entity**: `val beschreibung: Beschreibung? = null` (nullable with default)
2. **MongoProdukt**: `@param:BsonProperty("beschreibung") val beschreibung: String?`
3. **MongoDB→Domain**: `beschreibung = this.beschreibung?.let { Beschreibung.create(it) }`
4. **Domain→MongoDB**: `beschreibung = produkt.beschreibung?.asString()`
5. **Test helpers**: Updated erstelleProdukt(), created erstelleMongoProdukt()

## Patterns Discovered

### 1. Null-Safe Value Object Mapping Pattern

**Pattern**: Use elvis operator for optional value object fields
```kotlin
// MongoDB → Domain
beschreibung = this.beschreibung?.let { Beschreibung.create(it) }

// Domain → MongoDB
beschreibung = produkt.beschreibung?.asString()
```
- **Benefit**: Clean, idiomatic Kotlin for nullable transformations
- **Trade-off**: None - this is the standard Kotlin approach
- **When to use**: All optional value object fields in entity/adapter mapping

### 2. Test Helper Parameter Pattern

**Pattern**: Add optional parameters to test helpers with default null
```kotlin
fun erstelleProdukt(
  // ... existing parameters
  beschreibung: String? = null,
) = Produkt(
  // ... existing fields
  beschreibung = beschreibung?.let { Beschreibung.create(it) },
)
```
- **Benefit**: Backward compatibility for existing tests, explicit beschreibung in new tests
- **Trade-off**: Slightly longer parameter list (Suppressed with @Suppress("LongParameterList"))
- **When to use**: Test data builders for entities with optional fields

### 3. Mapping Test Separation Pattern

**Pattern**: Create dedicated *MappingTest.kt files for adapter logic
- **Benefit**: Clear separation between entity tests (ProduktTest) and adapter tests (MongoProduktMappingTest)
- **Clarity**: Isolated testing of nachProdukt()/vonProdukt() transformation logic
- **When to use**: When adapter has complex bidirectional mapping logic

### 4. Backward Compatibility Test Pattern

**Pattern**: Explicit integration tests for null handling and existing data
```kotlin
@Test
internal fun `ProduktMongoRepository speichert und lädt Produkt ohne Beschreibung für Rückwärtskompatibilität`() {
  val produkt = erstelleProdukt(beschreibung = null)
  repository.speicherProdukt(produkt)
  // ... verify beschreibung is null after round-trip
}
```
- **Benefit**: Validates migration safety for production MongoDB documents
- **Critical**: Ensures existing data without beschreibung field can be loaded

2. **Test Helper Organization**: TestProduktHelper.kt and TestMongoProduktHelper.kt could be consolidated
3. **MongoDB Null Handling**: No explicit tests for corrupt data (empty string in MongoDB) - should add

---

## Anti-Patterns to Avoid

1. **Nullable Value Objects Without Default**: Always provide `= null` default for backward compatibility
2. **Forgetting Null-Safe Operators**: Use `?.let {}` pattern consistently for optional transformations
3. **Skipping Backward Compatibility Tests**: Critical to test null handling in MongoDB round-trips
4. **Modifying Existing Tests**: New fields should not break existing test assertions
5. **Missing BsonProperty Annotation**: MongoDB fields must have `@param:BsonProperty("fieldName")` annotation
