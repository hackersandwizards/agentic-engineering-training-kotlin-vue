## Context

Implemented the `Beschreibung` value object as part of the Finden product search service domain model. This value object represents product descriptions with validation rules (max 500 characters, not blank) and XSS protection via HTML sanitization.

**Domain Context**:
- **Package**: `de.blume2000.finden.domain.model.produkte.produkt`

## Problem

**Requirements**:
1. Create immutable value object for product descriptions
2. Validate max length (500 characters)
3. Validate not blank (before AND after sanitization)
4. Sanitize HTML/XSS attempts
5. Follow DDD principles and existing patterns
6. Achieve 100/100 quality standard

**Challenges Encountered**:
1. **Java Version Mismatch**: Java 25 was active, project requires Java 21.0.8-tem
2. **Pattern Conflict**: ADR-0010 prescribes factory method pattern, existing code (Produktname) uses init block with `var`
3. **Detekt Violation**: Too many throw statements in single function (max 2)
4. **Immutability**: Initial implementation used `var` violating value object principles

---

## Solution

### Architecture Decisions

**Pattern Choice**: Factory Method with Private Constructor (ADR-0010 Compliant)
- **Rationale**: Despite existing init block pattern in Produktname, chose factory method for:
  - True immutability (`val` instead of `var`)
  - Clear separation of validation from construction
  - Compliance with documented architectural standard
  - Passes ArchUnit tests requiring final fields

**Key Design Elements**:
1. **Private constructor** with `private val value: String` (immutable)
2. **Factory method** `Beschreibung.create(rawValue: String)` for validation + sanitization
3. **Extracted validation methods** to satisfy Detekt ThrowsCount rule
4. **Convenience property** `length` for API usability
5. **Annotations**: `@ValueObject` (jMolecules) + `@ConsistentCopyVisibility` (Kotlin)

**Core Implementation** (Beschreibung.kt):
```kotlin
@ValueObject
@ConsistentCopyVisibility
data class Beschreibung private constructor(
  private val value: String
) {
  companion object {
    const val MAX_LENGTH = 500

    fun create(rawValue: String): Beschreibung {
      validatePreSanitization(rawValue)
      val sanitizedValue = rawValue.sanitize()
      validatePostSanitization(sanitizedValue)
      return Beschreibung(sanitizedValue)
    }

    private fun validatePreSanitization(value: String) {
      require(value.isNotBlank()) {
        throw BeschreibungIstLeerException("Beschreibung darf nicht leer sein: '$value'")
      }
    }

    private fun validatePostSanitization(value: String) {
      require(value.isNotBlank()) {
        throw BeschreibungIstLeerException("Beschreibung ist nach Sanitization leer")
      }
      require(value.length <= MAX_LENGTH) {
        throw BeschreibungIstZuLangException(
          "Beschreibung überschreitet Maximum von $MAX_LENGTH Zeichen: ${value.length} Zeichen"
        )
      }
    }
  }

  fun asString(): String = value
  val length: Int get() = value.length
}
```

## Patterns Discovered

### 1. Factory Method Pattern for Value Objects

**Pattern**: Private constructor + companion object factory method
- **Benefit**: Separates validation from construction, enables pre-construction transformations (sanitization)
- **Trade-off**: More verbose than init block, requires explicit factory call
- **When to use**: Complex validation, transformations before construction, need for true immutability

### 2. Double Blank Validation Pattern

**Pattern**: Validate blank BEFORE and AFTER sanitization
```kotlin
validatePreSanitization(rawValue)  // Check: empty/whitespace
val sanitized = rawValue.sanitize()
validatePostSanitization(sanitized)  // Check: empty after HTML removal
```
- **Benefit**: Prevents HTML-only input (`<div></div>`) from creating empty value objects
- **Security**: Critical for XSS prevention (ADR-0025)

### 3. Validation Method Extraction Pattern

**Pattern**: Extract validation into separate private methods to satisfy linting rules
- **Benefit**: Satisfies Detekt ThrowsCount rule (max 2 throws per function)
- **Clarity**: Named validation methods document intent (`validatePreSanitization`, `validatePostSanitization`)

### 4. Domain Exception Hierarchy

**Pattern**: Custom exceptions inheriting from `ProdukteException` base class
- **Benefit**: Domain-specific error types, German ubiquitous language
- **Usage**: `BeschreibungIstLeerException`, `BeschreibungIstZuLangException`

## Lessons Learned

### What Went Well

1. **Systematic TDD Workflow**: Workflow-orchestrator subagent provided clear phase-by-phase guidance
2. **Parallel Research**: Running documentation-researcher, pattern-analyzer, and architecture-advisor simultaneously saved significant time
3. **Quality Gates**: Incremental validation (tests → detekt → build) caught issues early
4. **Refactoring**: Transitioning from 85/100 to 100/100 was straightforward with existing test coverage

### What Could Be Improved

1. **Java Version Setup**: Should verify Java version at project start, not after first compile failure
2. **Pattern Consistency**: Codebase has conflicting patterns (Produktname uses init blocks vs ADR-0010 factory methods) - needs architectural alignment
3. **Detekt Rules**: ThrowsCount rule should be documented earlier to avoid refactoring rework

### Technical Debt Identified

1. **Produktname Refactoring**: Existing `Produktname` uses `var` + init block pattern, should be refactored to factory method for consistency
2. **ArchUnit Gaps**: ArchUnit doesn't enforce factory method pattern, only field immutability - could strengthen architecture tests
3. **ADR Update**: Either update ADR-0010 to reflect actual patterns OR refactor all value objects to match ADR

---

## Anti-Patterns to Avoid

1. **Mutable Value Objects**: Using `var` in value objects violates DDD principles and data class equality semantics
2. **Constructor Parameter Mutation**: Mutating constructor parameters (`value = value.sanitize()`) is surprising and non-idiomatic
3. **Missing Post-Sanitization Validation**: Always validate AFTER sanitization to catch HTML-only input
4. **Too Many Throws**: Keep throw statements per function ≤ 2 (Detekt rule) by extracting validation methods
5. **Public Constructor with Complex Logic**: Use private constructor + factory method for complex validation
