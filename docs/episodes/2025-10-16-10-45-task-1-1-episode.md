## Context

Task 1.1 is a subtask of Task 1 (Beschreibung Value Object), which is part of the Finden SCS (product search service) domain model implementation using Domain-Driven Design (DDD) principles in Kotlin/Quarkus.

The objective is to create a basic `Beschreibung` (Product Description) value object following the established patterns in the codebase while adhering to CUPID principles and architectural standards.

---

## Problem Statement

**Challenge**: Implement a new DDD value object (`Beschreibung`) that:
1. Follows established patterns from similar value objects (Produktname, ProduktbildUrl, Produktfarbe)
2. Enforces business invariants (non-empty, max length 500 chars after sanitization)
3. Includes security measures (HTML sanitization to prevent XSS attacks)
4. Maintains immutability and value-based equality
5. Complies with architecture tests and CUPID principles
6. Uses German domain language (Ubiquitous Language)

**Questions During Implementation**:
- Should validation use init block or factory method pattern?
- How to handle blank values after HTML sanitization?
- What should be the exact max length constraint?
- Where should exception classes be placed?

### Implementation Details

```kotlin
@ValueObject
data class Beschreibung(
  private var value: String
) {
  init {
    if (value.isBlank()) {
      throw BeschreibungIstLeerException("Beschreibung darf nicht leer sein")
    }
    value = value.sanitize()
    if (value.isBlank()) {
      throw BeschreibungIstLeerException("Beschreibung darf nach Sanitization nicht leer sein")
    }
    if (value.length > 500) {
      throw BeschreibungIstZuLangException("Beschreibung darf maximal 500 Zeichen lang sein, ist aber ${value.length}")
    }
  }

  fun asString(): String = value
}
```

### Design Decisions

| Decision | Rationale | Alternatives Considered |
|----------|-----------|--------------------------|
| **Init block validation** | Consistent with existing value objects (Produktname, Produktnummer); simpler than factory methods; clear validation flow | Factory method pattern (ADR-0010 recommended); both are valid and compliant |
| **Dual blank checks** | Catches HTML-only content that becomes empty after sanitization; prevents storing meaningless data | Single blank check (less secure); check only after sanitization (misses HTML-only inputs) |
| **500 char max length** | ADR-0010 example specifies this; reasonable for product descriptions; prevents database bloat | Configurable constant; per-deployment limits; no limit (risky) |
| **German naming** | Aligns with ubiquitous language (business domain speaks German); project standard | English naming (violates project language policy); English + German (inconsistent) |
| **data class** | Provides automatic equals/hashCode/toString; value-based equality semantics; DDD best practice | Manual implementation; sealed classes; regular classes |
| **@ValueObject annotation** | jMolecules DDD support; enables architecture tests; makes pattern explicit; tooling support | No annotation (loses validation); custom annotation (reinventing wheel) |

## Patterns Discovered

### 1. Value Object Pattern (Established in Codebase)

**Pattern**: DDD Value Object with init block validation

**Evidence**: Found in Produktname.kt, Produktnummer.kt, Produktfarbe.kt

**Key Characteristics**:
- `@ValueObject` annotation from jMolecules (org.jmolecules.ddd)
- `data class` for automatic value semantics
- Private mutable `var value: String` field
- Validation in init block with early throw
- HTML sanitization via `String.sanitize()` extension
- Public `asString(): String` accessor

**Conformance**: ✅ Beschreibung follows this pattern exactly

### 2. Exception Hierarchy Pattern

**Pattern**: Domain-specific exceptions extending ProdukteException

**Evidence**: ProduktnameIstLeerException, ProduktnummerIstLeerException, ProduktfarbeEnthältUpperCaseZeichenException

**Naming Convention**: `{ValueObjectName}Ist{Condition}Exception` (German)

**Conformance**: ✅ BeschreibungIstLeerException and BeschreibungIstZuLangException follow pattern

### 3. Test Pattern (JUnit 5 + Strikt + BDD)

**Pattern**: German backtick-quoted test names with Given-When-Then structure

**Evidence**: ProduktnameTest.kt, ProduktfarbeTest.kt

**Characteristics**:
- `@Tag("unit")` for test classification
- `expectThat()` and `expectThrows()` from Strikt library
- German language in test descriptions
- Clear Given-When-Then flow

**Conformance**: ✅ BeschreibungTest follows this pattern

### 4. Package Structure Pattern

**Pattern**: Domain model value objects in `de.blume2000.finden.domain.model.produkte.produkt`

**Evidence**: All 15+ value objects in this package

**Architecture**: Onion/Hexagonal - innermost domain layer

**Conformance**: ✅ Beschreibung.kt placed in correct package

## Lessons Learned

### What Went Well ✅

1. **Pattern Consistency**: Existing codebase patterns (Produktname, Produktnummer) made implementation straightforward
2. **Documentation**: ADRs (especially ADR-0010, ADR-0025, ADR-0028) provided clear guidance
3. **Architecture Tests**: ValueObjectTest.kt enforces patterns automatically - no need for manual verification
4. **Test-First Approach**: BDD tests clearly defined requirements before implementation
5. **Dual Blank Checks**: Catching both pre- and post-sanitization blanks prevents security gaps

### What Could Be Improved 🔧

1. **Factory Method vs Init Block**: Both patterns are valid; could add documentation explaining when to choose each
2. **Build Environment**: Gradle build requires Aiven schema registry environment variables - could document this in README
3. **Token Limits**: Quality-assurance-expert subagent hit token limits - could use more focused prompts
4. **Exception Message Detail**: Could include sanitized vs unsanitized value in exception messages for debugging

### Anti-Patterns to Avoid ❌

1. ❌ **Public fields**: Must keep `value` field private for encapsulation
2. ❌ **Mutable state**: Although using `private var`, it's only mutated in init block; effectively immutable
3. ❌ **Missing sanitization**: XSS prevention is critical - must sanitize before validation
4. ❌ **Single validation check**: Must check blank BOTH before and after sanitization
5. ❌ **English naming**: Must use German domain language (project standard)
6. ❌ **Skip architecture annotations**: @ValueObject annotation is mandatory for architecture test compliance
7. ❌ **No custom exceptions**: Domain exceptions are essential for caller error handling
